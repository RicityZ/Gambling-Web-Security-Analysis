## 🎯 สรุปผู้บริหาร

โครงการนี้นำเสนอการวิเคราะห์ความปลอดภัยอย่างครบถ้วนของแอปพลิเคชันเว็บพนันออนไลน์ โดยแสดงช่องโหว่ที่สำคัญและวิธีการแก้ไข การศึกษานี้ดำเนินการเป็นส่วนหนึ่งของ **โครงงานไซเบอร์ซีเคียวริตี้ รหัส 101203**

### ผลการค้นพบที่สำคัญ:

| ช่องโหว่ | ความรุนแรง | สถานะ |
|---------|----------|--------|
| Client-Side Injection (MITM) | 🔴 **วิกฤต** | ✅ แก้ไขแล้ว |
| Broken Business Logic | 🔴 **วิกฤต** | ✅ แก้ไขแล้ว |
| การตรวจสอบข้อมูลไม่เพียงพอ | 🟡 **สูง** | ✅ แก้ไขแล้ว |
| การควบคุมการเงินฝั่ง Client | 🔴 **วิกฤต** | ✅ แก้ไขแล้ว |

### ผลกระทบ:
- **ระบบที่มีช่องโหว่**: ผู้โจมตีสามารถปรับแต่งจำนวนเงินที่ชนะเพื่อรับเงินเสมือนได้ไม่จำกัด
- **ระบบที่แก้ไขแล้ว**: การตรวจสอบและการอนุญาตฝั่งเซิร์ฟเวอร์อย่างครบถ้วนป้องกันการโจมตีทุกรูปแบบที่ทดสอบ

## 🎓 ภาพรวมโครงการ

### วัตถุประสงค์

> **เพื่อศึกษาและจำลองการโจมตีเว็บพนันออนไลน์ผ่านช่องโหว่ทางไซเบอร์**

เพื่อศึกษาและจำลองการโจมตีเว็บไซต์พนันออนไลน์ผ่านช่องโหว่ด้านไซเบอร์ซีเคียวริตี้ โดยมีเป้าหมายในการทำความเข้าใจทิศทางการโจมตีและการนำมาตรการรักษาความปลอดภัยที่แข็งแกร่งมาใช้

### ขอบเขต

- **แอปพลิเคชันเป้าหมาย**: เกมสล็อตไพ่นกกระจอกแบบเว็บ
- **ช่องทางการโจมตี**: การฉีดค่าฝั่ง Client, การโจมตีแบบ Man-in-the-Middle (MITM)
- **สภาพแวดล้อมการทดสอบ**: สภาพแวดล้อมพัฒนาที่แยกออกมา
- **ระยะเวลา**: [ระยะเวลาโครงการ]

### เทคโนโลยีที่ใช้

**แอปพลิเคชันที่มีช่องโหว่:**
- Node.js + Express.js (Backend)
- Vanilla JavaScript (Frontend)
- HTML5 + CSS3 (UI)

**เครื่องมือโจมตี:**
- mitmproxy (การดักจับ Traffic)
- Python (การทำงานอัตโนมัติของสคริปต์โจมตี)
- Browser DevTools (การวิเคราะห์)

**แอปพลิเคชันที่ปลอดภัย:**
- สถาปัตยกรรมที่เซิร์ฟเวอร์เป็นผู้ควบคุม
- การตรวจสอบและการกรองข้อมูลนำเข้า
- แนวปฏิบัติด้านบัญชีที่ปลอดภัย

## 🔍 การวิเคราะห์ช่องโหว่

### 1. การฉีดค่าฝั่ง Client (การโจมตี MITM)

**การจำแนกแบบ CVE:** CWE-345 (การตรวจสอบความถูกต้องของข้อมูลไม่เพียงพอ)

#### คำอธิบาย:
แอปพลิเคชันที่มีช่องโหว่ยอมรับค่า `totalWin` จาก Client โดยไม่มีการตรวจสอบฝั่งเซิร์ฟเวอร์ ทำให้ผู้โจมตีสามารถใช้เทคนิค Man-in-the-Middle (MITM) เพื่อดักจับและแก้ไข HTTP requests/responses ได้

#### โค้ดที่มีช่องโหว่ (vulnerable-version/server.js):
```javascript
app.post('/spin', (req, res) => {
  const { bet, totalWin: injectedWin } = req.body  // ❌ เชื่อถือข้อมูลจาก Client
  
  if (typeof injectedWin === 'number' && injectedWin > 0) {
    totalWins++
    totalWin = injectedWin  // ❌ ไม่มีการตรวจสอบ!
    balance += (totalWin - bet)  // ❌ Client ควบคุมยอดเงินได้
  }
  // ...
})
```

#### วิธีการใช้ประโยชน์:
ใช้ **mitmproxy** เพื่อดักจับ Traffic:

```python
# vulnerable-version/demo hack/inject_win.py
def request(flow: http.HTTPFlow):
    if "spin" in flow.request.pretty_url:
        data = json.loads(flow.request.text)
        data["totalWin"] = 9999  # ✅ ฉีดค่าตามต้องการ
        flow.request.text = json.dumps(data)
```

#### ผลกระทบ:
- ⚠️ ผู้โจมตีได้รับเงินเสมือนไม่จำกัด
- ⚠️ ข้ามผ่านตรรกะเกมทั้งหมด
- ⚠️ ละเมิดตรรกะทางธุรกิจ
- ⚠️ ความสมบูรณ์ทางการเงินถูกทำลาย

### 2. ตรรกะทางธุรกิจที่ผิดพลาด

**การจำแนกแบบ CVE:** CWE-840 (ข้อผิดพลาดทางตรรกะธุรกิจ)

#### คำอธิบาย:
ตรรกะทางธุรกิจของแอปพลิเคชันสมมติว่า Client จะรายงานผลลัพธ์เกมอย่างซื่อสัตย์ ข้อบกพร่องทางสถาปัตยกรรมพื้นฐานนี้ทำให้ผู้โจมตีสามารถข้ามกลไกเกมที่ตั้งใจไว้

#### พื้นที่ปัญหา:

**การคำนวณที่ควบคุมโดย Client:**
```javascript
// vulnerable-version/main.js
async function runCascade() {
  // ...
  totalWin += winAmount;  // Client คำนวณการชนะ
}

fetch('http://127.0.0.1:3001/spin', {
  body: JSON.stringify({ grid, bet, totalWin })  // ❌ ส่งค่าจาก Client
})
```

**เซิร์ฟเวอร์ยอมรับอย่างไร้เงื่อนไข:**
```javascript
// เซิร์ฟเวอร์ใช้ totalWin จาก Client โดยไม่คำนวณใหม่
balance += (totalWin - bet)  // ❌ เชื่อการคำนวณของ Client
```

#### ผลกระทบ:
- ⚠️ ไม่มีการแยกระหว่างการแสดงผลของ Client และอำนาจของเซิร์ฟเวอร์
- ⚠️ ไม่สามารถตรวจจับการโกงได้
- ⚠️ บันทึกการตรวจสอบไม่มีความหมาย
- ⚠️ ไม่สามารถรับประกันการเล่นที่ยุติธรรมได้

---

### 3. การตรวจสอบข้อมูลนำเข้าไม่เพียงพอ

**การจำแนกแบบ CVE:** CWE-20 (การตรวจสอบข้อมูลนำเข้าที่ไม่เหมาะสม)

#### คำอธิบาย:
เวอร์ชันที่มีช่องโหว่ขาดการตรวจสอบข้อมูลนำเข้า ทำให้:
- จำนวนเงินเดิมพันติดลบได้
- เดิมพันมากกว่ายอดเงินคงเหลือ
- ส่งสถานะเกมที่ไม่ถูกต้องได้
- ข้อมูลตารางเกมที่ผิดรูปแบบ

#### การตรวจสอบที่ขาดหายไป:
```javascript
// ❌ ไม่มีการตรวจสอบสำหรับ:
// - ช่วงจำนวนเงินเดิมพัน
// - ยอดเงินคงเหลือเพียงพอ
// - ความถูกต้องของโครงสร้างตาราง
// - ความถูกต้องของประเภทข้อมูล
```

#### ขั้นตอนที่ 4: ดำเนินการโจมตี
1. เปิด `http://localhost:3001`
2. วางเดิมพันจำนวนใดก็ได้และคลิก **SPIN**
3. สังเกตการฉีดผลชนะ **9999 บาท**

### ผลการโจมตี

**ก่อนการโจมตี:**
```
ยอดเงิน: 10,000 บาท
เดิมพัน: 100 บาท
```

**หลังการหมุน 1 ครั้งที่ถูกแก้ไข:**
```
ยอดเงิน: 19,899 บาท (+9,799 บาท)
จำนวนเงินที่ชนะ: 9,999 บาท (ถูกฉีด)
```

**หลังการหมุน 10 ครั้งที่ถูกแก้ไข:**
```
ยอดเงิน: 108,890 บาท
กำไร: 98,890 บาท จากยอดเริ่มต้น 10,000 บาท
```

### หลักฐานภาพ

ดูที่โฟลเดอร์ [docs/](./docs/) สำหรับภาพหน้าจอที่แสดง:
- mitmproxy ดักจับ requests
- ค่า `totalWin` ที่ถูกแก้ไข
- การเปลี่ยนแปลงยอดเงินที่ถูกจัดการ
- บันทึกของเซิร์ฟเวอร์ที่แสดงค่าที่ถูกฉีด

## 🔒 การแก้ไขความปลอดภัย

### วิธีแก้ไข: สถาปัตยกรรมที่เซิร์ฟเวอร์เป็นผู้ควบคุม

เวอร์ชันที่แก้ไขแล้วนำโมเดล **ไม่เชื่อถือ Client เลย** มาใช้ โดยเซิร์ฟเวอร์ควบคุมตรรกะเกมและการคำนวณทางการเงินทั้งหมด

### มาตรการรักษาความปลอดภัยที่สำคัญ

#### 1. ✅ ตรรกะเกมที่เซิร์ฟเวอร์เป็นผู้ควบคุม

**ก่อน (มีช่องโหว่):**
```javascript
// Client คำนวณ, เซิร์ฟเวอร์ยอมรับ
const totalWin = clientCalculation()
sendToServer({ totalWin })  // ❌
```

**หลัง (ปลอดภัย):**
```javascript
// Client ส่งเฉพาะสถานะเกม
sendToServer({ bet, grid })  // ✅ ไม่มีค่าที่คำนวณแล้ว

// เซิร์ฟเวอร์คำนวณทุกอย่าง
const totalWin = serverCalculateWin(grid)  // ✅ อำนาจของเซิร์ฟเวอร์
```

#### 2. ✅ การตรวจสอบข้อมูลนำเข้าอย่างครบถ้วน

```javascript
// patched-version/server.js
app.post('/spin', (req, res) => {
  const { bet, grid } = req.body
  
  // ตรวจสอบจำนวนเงินเดิมพัน
  if (typeof bet !== 'number' || bet < 1 || bet > 1000) {
    return res.status(400).json({ error: 'จำนวนเงินเดิมพันไม่ถูกต้อง' })
  }
  
  // ตรวจสอบยอดเงินคงเหลือก่อนดำเนินการ
  if (balance < bet) {
    return res.status(400).json({ error: 'ยอดเงินคงเหลือไม่เพียงพอ' })
  }
  
  // ตรวจสอบโครงสร้างตาราง
  if (!validateGrid(grid)) {
    return res.status(400).json({ error: 'ข้อมูลตารางไม่ถูกต้อง' })
  }
  
  // ... ดำเนินการต่อด้วยข้อมูลที่ตรวจสอบแล้ว
})
```

#### 3. ✅ กระบวนการบัญชีที่ปลอดภัย: "หักก่อน-จ่ายหลัง"

**หลักการ:** หักจำนวนเงินเดิมพันก่อนเสมอก่อนคำนวณเงินรางวัล เพื่อป้องกัน race conditions และการจัดการ

```javascript
// ✅ ปลอดภัย: หักก่อน
balance -= bet

// คำนวณเงินรางวัล (ฝั่งเซิร์ฟเวอร์)
const serverCalculatedWin = calculateTotalWin(grid)

// จ่ายเงินรางวัลเฉพาะเมื่อถูกต้อง
if (isWin && serverCalculatedWin > 0) {
  balance += serverCalculatedWin  // ✅ จ่ายหลังการตรวจสอบ
}
```

**ทำไมสิ่งนี้จึงสำคัญ:**
- ป้องกันช่องโหว่ยอดเงินติดลบ
- รับประกันธุรกรรมแบบ atomic
- ร่องรอยการตรวจสอบถูกต้อง
- ไม่มีทางจัดการยอดเงินได้

#### 4. ✅ การตรวจสอบและการคำนวณตารางใหม่

```javascript
function validateGrid(grid) {
  if (!Array.isArray(grid) || grid.length !== 4) return false
  
  for (let row of grid) {
    if (!Array.isArray(row) || row.length !== 5) return false
    for (let tile of row) {
      if (!(tile in tileValues)) return false  // ✅ เฉพาะไพ่ที่ถูกต้อง
    }
  }
  return true
}

function calculateTotalWin(grid) {
  let totalScore = 0
  let multiplier = 1
  
  // เซิร์ฟเวอร์คำนวณการจับคู่และการตกลงทั้งหมดใหม่
  while (hasMatch) {
    const matched = findMatches(gridCopy)
    const score = removeMatchesAndScore(gridCopy, matched)
    if (score > 0) {
      totalScore += score * multiplier  // ✅ เซิร์ฟเวอร์ควบคุมการคำนวณ
      multiplier = getNextMultiplier(multiplier)
      collapseGrid(gridCopy)
    } else {
      hasMatch = false
    }
  }
  
  return totalScore  // ✅ ค่าที่เซิร์ฟเวอร์คำนวณ
}
```

✅ **ความปลอดภัยของเว็บแอปพลิเคชัน**
- การระบุและวิเคราะห์ช่องโหว่
- การทำแผนที่ช่องทางการโจมตี
- การนำการควบคุมความปลอดภัยมาใช้

✅ **การทดสอบการเจาะระบบ**
- การดำเนินการโจมตี MITM
- การดักจับและแก้ไข Traffic
- การพัฒนา Exploit

✅ **การพัฒนาที่ปลอดภัย**
- การตรวจสอบฝั่งเซิร์ฟเวอร์
- กลยุทธ์การป้องกันแบบหลายชั้น
- แนวปฏิบัติการเขียนโค้ดที่ปลอดภัย

✅ **การพัฒนา Full-Stack**
- การออกแบบ Backend API
- ขอบเขตความปลอดภัยระหว่าง Frontend-Backend
- การจัดการ State

### หลักการรักษาความปลอดภัยที่นำมาใช้

1. **สถาปัตยกรรม Zero Trust** - ไม่ไว้วางใจข้อมูลจาก Client
2. **การป้องกันแบบหลายชั้น** - การตรวจสอบหลายระดับ
3. **สิทธิ์ขั้นต่ำ** - Client มีเพียงสิทธิ์แสดงผล
4. **ปลอดภัยตั้งแต่การออกแบบ** - ความปลอดภัยสร้างเข้าไป ไม่ใช่ติดเพิ่มทีหลัง

## ⚖️ ข้อพิจารณาด้านจริยธรรม

### การเปิดเผยอย่างมีความรับผิดชอบ

งานวิจัยนี้ดำเนินการ:

✅ **ในสภาพแวดล้อมที่ควบคุมได้** - แอปพลิเคชันที่โฮสต์เองเท่านั้น  
✅ **เพื่อการศึกษา** - การเรียนรู้และการสอนด้านความปลอดภัย  
✅ **พร้อมการแก้ไขครบถ้วน** - มีวิธีแก้ไขสำหรับช่องโหว่ทั้งหมด  
✅ **ด้วยเจตนาที่ถูกต้อง** - เพื่อป้องกันการใช้ประโยชน์ ไม่ใช่เพื่อเปิดโอกาส

### ประกาศทางกฎหมาย

⚠️ **คำเตือน**: การใช้เทคนิคเหล่านี้กับระบบที่ใช้งานจริงโดยไม่ได้รับอนุญาตเป็นสิ่งผิดกฎหมายและผิดจริยธรรม

**ห้าม:**
- โจมตีระบบที่คุณไม่ได้เป็นเจ้าของ
- ใช้เทคนิคเหล่านี้เพื่อประโยชน์ส่วนตัว
- ใช้ประโยชน์จากช่องโหว่โดยไม่ได้รับอนุญาต
- แจกจ่ายเครื่องมือ Exploit อย่างเป็นอันตราย

**ควร:**
- เรียนรู้จากงานวิจัยนี้
- นำหลักการรักษาความปลอดภัยมาใช้กับโครงการของคุณเอง
- ดำเนินการทดสอบความปลอดภัยที่ได้รับอนุญาต
- ส่งเสริมแนวปฏิบัติการพัฒนาที่ปลอดภัย

### คำแถลงวัตถุประสงค์ทางการศึกษา

โครงการนี้มีจุดมุ่งหมายเพื่อ:

- **ให้ความรู้** นักพัฒนาเกี่ยวกับช่องโหว่ด้านความปลอดภัยทั่วไป
- **สาธิต** ความสำคัญของการตรวจสอบฝั่งเซิร์ฟเวอร์
- **ส่งเสริม** การแฮ็กอย่างมีจริยธรรมและการเปิดเผยอย่างมีความรับผิดชอบ
- **หักล้าง** ความเข้าใจผิดเกี่ยวกับระบบพนันออนไลน์

ตามที่ระบุไว้ในวัตถุประสงค์ของโครงการ:

> _"ช่วยหักล้างภาพฝันว่าเล่นแล้วรวย"_

ระบบพนันออนไลน์—แม้แต่ที่ถูกกฎหมาย—ก็มี house edge และที่ไม่ถูกกฎหมายอาจมีช่องโหว่ที่ใช้ประโยชน์ได้หรือเป็นการฉ้อโกงโดยตรง การวิจัยนี้เปิดเผยความจริงทางเทคนิคเบื้องหลังอินเทอร์เฟซที่ฉูดฉาด

|---------|-------------------|-----------------|
| **การคำนวณการชนะ** | ❌ ฝั่ง Client | ✅ ฝั่งเซิร์ฟเวอร์ |
| **การตรวจสอบข้อมูลนำเข้า** | ❌ ไม่มี | ✅ ครบถ้วน |
| **การตรวจสอบยอดเงิน** | ❌ หลังธุรกรรม | ✅ ก่อนธุรกรรม |
| **การตรวจสอบตาราง** | ❌ ไม่ตรวจสอบ | ✅ ตรวจสอบเต็มที่ |
| **การบันทึกการตรวจสอบ** | ❌ น้อยมาก | ✅ ครบถ้วน |
| **การโจมตี MITM** | ❌ สำเร็จ | ✅ ถูกบล็อก |
| **การจัดการคำขอ** | ❌ ใช้ประโยชน์ได้ | ✅ ถูกปฏิเสธ |
| **ความสมบูรณ์ทางการเงิน** | ❌ ถูกทำลาย | ✅ ได้รับการปกป้อง |

## 🎓 แหล่งข้อมูลการเรียนรู้

### หัวข้อที่เกี่ยวข้องที่ควรศึกษา:

- **OWASP Top 10** - ความเสี่ยงด้านความปลอดภัยของเว็บแอปพลิเคชัน
- **CWE/SANS Top 25** - จุดอ่อนของซอฟต์แวร์ที่อันตรายที่สุด
- **การโจมตี MITM** - เทคนิคการโจมตีแบบ Man-in-the-Middle
- **การตรวจสอบข้อมูลนำเข้า** - แนวปฏิบัติการจัดการข้อมูลนำเข้าที่ปลอดภัย
- **การรับรองความถูกต้องและการอนุญาต** - การจัดการข้อมูลประจำตัวและการเข้าถึง

### เครื่องมือที่แนะนำ:

- **mitmproxy** - Proxy HTTPS แบบโต้ตอบ
- **Burp Suite** - เครื่องสแกนช่องโหว่เว็บ
- **OWASP ZAP** - เครื่องมือทดสอบความปลอดภัย
- **Postman** - การทดสอบ API
## 📄 ใบอนุญาต

โครงการนี้เผยแพร่ **เพื่อการศึกษาเท่านั้น**

**ข้อกำหนด:**
- ✅ ใช้เพื่อการศึกษาและเรียนรู้ได้ฟรี
- ✅ อาจใช้ใน Portfolio ทางวิชาการ
- ✅ สามารถอ้างอิงในการวิจัยด้านความปลอดภัย
- ❌ ไม่ใช้สำหรับการทดสอบความปลอดภัยที่ไม่ได้รับอนุญาต
- ❌ ไม่ใช้เพื่อการค้า
- ❌ ไม่ใช้เพื่อวัตถุประสงค์ที่เป็นอันตราย


## 📈 การปรับปรุงในอนาคต

การปรับปรุงที่อาจเป็นไปได้ในการสำรวจ:

- [ ] เพิ่มระบบการรับรองความถูกต้อง JWT
- [ ] นำ rate limiting มาใช้
- [ ] เพิ่มการป้องกัน CSRF
- [ ] วิเคราะห์ความปลอดภัย WebSocket
- [ ] เสริมความแข็งแกร่งการจัดการ Session
- [ ] ความปลอดภัยของฐานข้อมูล (การป้องกัน SQL injection)
- [ ] การป้องกัน XSS
- [ ] Content Security Policy (CSP)
